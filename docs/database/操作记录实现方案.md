# 操作记录实现方案

## 1. 目标
- 为平台管理操作建立可审计、可检索、可追责的证据链。
- 满足安全合规、问题追溯、风控分析三类场景。

## 2. 采集架构
- 请求入口：`HandlerInterceptor` 记录请求上下文（IP、UA、TraceId、耗时）。
- 业务动作：AOP 注解 `@OperationAudit` 记录模块、动作、对象、结果。
- 异常补偿：全局异常处理器统一补写失败操作记录。
- 异步落库：通过内存队列或消息队列异步写入，避免阻塞主链路。

## 3. 字段规范
- 基础字段：`user_id`、`module_code`、`action_code`、`target_id`、`created_at`
- 上下文字段：`trace_id`、`request_ip`、`user_agent`、`request_path`、`request_method`
- 数据字段：`request_json`、`response_json`、`before_json`、`after_json`
- 结果字段：`result_status`（SUCCESS/FAIL）、`error_code`、`error_msg`、`duration_ms`
- 治理字段：`risk_level`（LOW/MEDIUM/HIGH）、`tenant_id`

## 4. 数据变更快照强制规则
- 适用范围：所有新增（INSERT）、修改（UPDATE）、删除（DELETE）类操作。
- 新增操作：`before_json` 为空，`after_json` 记录新增后完整业务对象（脱敏后）。
- 删除操作：`before_json` 记录删除前完整业务对象（脱敏后），`after_json` 为空。
- 修改操作：`before_json` 与 `after_json` 均必填，至少包含变更字段及主键。
- 失败回滚：若事务失败未落库，仍记录尝试修改的 `before_json` 与意图 `after_json`，并标记 `result_status=FAIL`。
- 脱敏要求：快照中的密码、密钥、token、证件号、手机号等字段必须脱敏或哈希。

## 5. 数据存储与生命周期
- 在线存储：MySQL（近 90 天）用于高频检索。
- 冷归档：对象存储或归档库（90 天后自动归档）。
- 保留策略：默认 180 天，可按合规要求延长。
- 清理策略：仅允许归档，不允许直接物理删除。

## 6. 检索与分析能力
- 多维检索：按用户、模块、动作、对象、时间范围、结果状态。
- 高风险筛选：仅看 `risk_level=HIGH` 或失败操作。
- 导出能力：按工单导出 CSV/Excel，导出动作记录二次审计。
- 趋势分析：失败率、敏感操作量、越权尝试次数趋势。

## 7. 安全要求
- 对密码、密钥、token、手机号等敏感字段做脱敏或哈希摘要。
- 仅审计角色可查看完整日志，普通管理员仅可看摘要。
- 导出文件加水印（导出人、时间、工单号）。

## 8. 表结构落地建议
- 主表：`operation_logs`（已存在，补充上下文与结果字段）。
- 明细表：`operation_log_details`（大字段分离，降低主表 IO 压力）。
- 索引：
  - `idx_operation_logs_user_time (user_id, created_at)`
  - `idx_operation_logs_trace_id (trace_id)`
  - `idx_operation_logs_module_action_time (module_code, action_code, created_at)`

## 9. 前端：公共操作日志抽屉（`OperationLogDrawer.vue`）

- 组件路径：`knot-front/src/components/common/OperationLogDrawer.vue`。
- 通过 **`title`**、**`drawerSize`**（默认 `60%`）、**`loadLogs()`**（返回 Promise 的加载函数）以及 **`showDescription` / `showEntityName` / `showOldNew`** 控制展示与数据拉取；打开抽屉时调用 `loadLogs()` 刷新列表。
- **枚举管理**：`loadLogs` 内调用 `listEnumOperationLogs(category)`（`/api/system/enums/operation-logs/{category}`），并设 `show-description=false`（不展示说明列）。
- **用户管理**：`loadLogs` 内调用 `listUserOperationLogs(userId)`（`/api/operation-logs/entity/User/{id}?module=user`）。
- 默认表格列含：时间、操作、对象、结果、说明（可关）、旧值/新值（可关）、错误。

## 10. 切面 SpEL 与旧/新值落库

- 编译启用 **`maven.compiler.parameters`**，方法参数名在切面中可作为 SpEL 变量（如 `#id`）。
- 另在 **`OperationLogAspect`** 中注册 **`#p0`、`#p1`…** 与参数位置对应，避免未保留参数名时 `#id` 解析失败导致 **`oldValueSpel` / `entityId` 为空**。
- 枚举更新/删除中旧值 SpEL 使用 **`@enumConfigService.getById(#p0)`**（`#p0` 为路径参数 `id`）。
- 用户变更使用 **`@userService.userAuditSnapshot(id)`** 生成不含密码的 Map JSON，避免把密码哈希写入日志。
- 异步落库失败时查看 **`OperationLogService`** 的 WARN 日志（原 `System.err` 已改为 SLF4J）。
- **`listByModule` / `listForEnumCategory` / `getById` 返回前**：若 `old_value` 与 `new_value` 均为 JSON **对象**，则按字段做 `JsonNode.equals` 比较，**仅保留值发生变化的字段**写入返回体（单方有值或非对象则原样返回）。
