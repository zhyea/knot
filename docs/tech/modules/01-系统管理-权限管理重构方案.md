# 系统管理权限重构方案

## 1. 目标与边界

### 1.1 目标
- 重建后台 `admin` 侧的模块、角色、用户权限管理能力。
- 统一菜单权限、页面权限、按钮权限、接口权限的模型。
- 后端采用 `@AuthCheck` 无参注解完成接口鉴权。
- 权限变更后可快速生效，不依赖重新发布前端或修改硬编码角色判断。

### 1.2 非目标
- 本方案不改造 `gateway` 鉴权。
- `gateway` 继续基于 `apikey`、消费者、路由规则、额度和频控运行。
- 不考虑兼容历史角色、历史菜单、历史权限表结构，允许推倒重建。

### 1.3 适用范围
- 仅适用于 `knot-server/knot-admin` 管理后台。
- 前端仅对管理后台菜单、页面和操作按钮做权限联动。

## 2. 现状问题

- 前端菜单目前主要是静态写死，无法根据权限动态下发。
- 前端登录态只保存 `roles`，缺少细粒度权限模型。
- 后端登录后只返回角色集合，没有真正的权限点集合。
- 接口侧只有认证，没有形成稳定的“接口权限绑定”模型。
- 角色目前更接近硬编码标签，而不是可配置的权限集合。

## 3. 总体设计

### 3.1 设计原则
- 后端鉴权以“权限点”而不是“角色编码”为核心。
- 角色只是权限集合，不直接承载业务逻辑。
- 菜单只是展示载体，不作为最终鉴权依据。
- 前端只负责可见性控制，后端负责最终授权控制。
- `@AuthCheck` 只表达“该接口需要鉴权”，不携带任何参数。

### 3.2 架构分层

#### 认证层
- 负责解析后台 JWT。
- 建立当前登录用户上下文。
- 仅判断“是否登录有效”。

#### 鉴权层
- 负责识别 `@AuthCheck`。
- 根据请求、控制器方法和数据库映射推导所需权限。
- 校验当前用户是否具备对应权限。

#### 权限模型层
- 负责维护模块、菜单、权限点、角色、用户之间的关系。
- 提供用户生效权限查询和菜单树构建能力。

#### 前端消费层
- 登录后读取“当前用户授权信息”。
- 动态构建菜单、页面访问能力和按钮显示能力。

## 4. 权限模型设计

### 4.1 模块 `module`
- 表示后台业务域，例如：
- `system`
- `model`
- `routing`
- `billing`
- 模块用于分组和展示，不直接参与最终鉴权。

### 4.2 菜单 `menu`
- 用于前端导航树和页面入口。
- 与路由、图标、排序、父子关系绑定。
- 菜单可见性由权限点控制，但菜单本身不是接口权限。

### 4.3 权限点 `permission`
- 权限点是鉴权最小单元。
- 建议支持以下类型：
- `MENU`：控制菜单显示
- `PAGE`：控制页面访问
- `ACTION`：控制按钮和局部操作
- `API`：控制接口访问

### 4.4 角色 `role`
- 角色是权限集合。
- 支持系统内置角色和自定义角色。
- 支持启用/禁用。
- 角色不应再作为前端写死判断依据。

### 4.5 用户 `user`
- 用户绑定一个或多个角色。
- 第一阶段不建议引入“用户直授权限”。
- 如后续确有需要，可在二期增加用户级权限覆盖。

## 5. 数据库设计

### 5.1 核心表

#### `sys_user`
- 用户基础信息。
- 关键字段：
- `id`
- `username`
- `password_hash`
- `real_name`
- `dept_id`
- `status`
- `authz_version`
- `last_login_time`

#### `sys_role`
- 角色定义。
- 关键字段：
- `id`
- `role_code`
- `role_name`
- `role_type`
- `built_in`
- `status`
- `remark`

#### `sys_user_role`
- 用户和角色的多对多关系。
- 关键字段：
- `user_id`
- `role_id`

#### `sys_module`
- 模块定义。
- 关键字段：
- `id`
- `module_code`
- `module_name`
- `icon`
- `sort_no`
- `status`

#### `sys_menu`
- 菜单定义。
- 关键字段：
- `id`
- `module_id`
- `parent_id`
- `menu_code`
- `menu_name`
- `route_path`
- `component_key`
- `icon`
- `sort_no`
- `status`

#### `sys_permission`
- 权限点定义。
- 关键字段：
- `id`
- `permission_code`
- `permission_name`
- `permission_type`
- `module_id`
- `status`
- `built_in`
- `remark`

#### `sys_role_permission`
- 角色和权限的多对多关系。
- 关键字段：
- `role_id`
- `permission_id`

#### `sys_api_permission_binding`
- 接口与权限点的绑定关系。
- 关键字段：
- `id`
- `permission_id`
- `http_method`
- `path_pattern`
- `controller_class`
- `handler_method`
- `status`

### 5.2 建议约束
- `sys_user.username` 唯一。
- `sys_role.role_code` 唯一。
- `sys_module.module_code` 唯一。
- `sys_menu.menu_code` 唯一。
- `sys_permission.permission_code` 唯一。
- `sys_api_permission_binding(http_method, path_pattern)` 唯一。

### 5.3 建议索引
- `sys_user_role(user_id)`
- `sys_user_role(role_id)`
- `sys_role_permission(role_id)`
- `sys_role_permission(permission_id)`
- `sys_api_permission_binding(http_method, path_pattern)`

## 6. 后端鉴权方案

### 6.1 注解设计

```java
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface AuthCheck {
}
```

说明：
- 注解不保留任何参数。
- 鉴权所需信息统一从请求、上下文和数据库中推导。

### 6.2 执行链路

1. `AdminAuthInterceptor`
- 解析后台 JWT。
- 构建当前用户认证上下文。

2. `AdminAuthorizationInterceptor`
- 判断控制器类或方法上是否存在 `@AuthCheck`。
- 提取当前请求信息：
- `httpMethod`
- `bestMatchingPattern`
- `controllerClass`
- `handlerMethod`

3. `PermissionResolveService`
- 根据 `httpMethod + bestMatchingPattern` 查询 `sys_api_permission_binding`。
- 得到当前接口需要的 `permission_code`。

4. `AuthorizationService`
- 查询当前用户生效权限集合。
- 判断是否具备目标权限。
- 无权限则抛统一异常。

### 6.3 推荐类设计

- `AdminAuthInterceptor`
- `AdminAuthorizationInterceptor`
- `CurrentAdminAuth`
- `AuthorizationService`
- `PermissionResolveService`
- `MenuAuthorizationService`
- `RolePermissionService`

### 6.4 JWT 建议
- JWT 中只保存最小身份信息：
- `userId`
- `username`
- `authzVersion`

不建议保存：
- 完整角色列表
- 完整权限列表

原因：
- 避免权限变更后 token 长时间不生效。
- 降低 token 与数据库状态不一致的问题。

### 6.5 缓存建议
- 缓存当前用户的生效权限集合。
- key 示例：
- `admin:authz:user:{userId}:v:{authzVersion}`

触发失效的场景：
- 用户角色变更
- 角色权限变更
- 用户状态变更
- 权限点或接口绑定关系变更

失效策略：
- 更新用户 `authz_version`
- 删除对应缓存

## 7. 前端联动方案

### 7.1 登录后加载
- 登录成功后调用 `/api/me/authorizations`。
- 返回内容建议包括：
- 当前用户信息
- 角色列表
- 权限码集合
- 动态菜单树

### 7.2 前端权限使用方式
- 页面路由通过菜单树动态注册。
- 页面访问通过 `PAGE` 权限控制。
- 按钮和行操作通过 `ACTION` 权限控制。

建议补充两个通用能力：
- `usePermission()`
- `v-permission`

示例：

```vue
<el-button v-permission="'system:user:create'">新建用户</el-button>
```

### 7.3 菜单构建原则
- 前端保留 `componentKey -> 组件` 的静态映射表。
- 后端只下发菜单元数据，不下发任意脚本路径。
- 避免形成前端可执行路径注入风险。

### 9.1 当前登录用户授权信息
- `GET /api/me/authorizations`

- `roles`
- `permissions`

## 10. 实施建议

### 第一阶段
- 重建表结构
- 接入 `@AuthCheck`
- 完成接口权限绑定
- 完成角色、权限、用户基础管理

### 第二阶段
- 前端动态菜单
- 按钮权限控制
- 权限缓存与权限变更快速失效

### 第三阶段
- 如确有需要，再补充：
- 用户直授权限
- 数据范围
- 审计与授权变更历史

## 11. 结论

后台 `admin` 侧推荐采用“无参 `@AuthCheck` + 数据库存储接口权限映射 + 用户角色权限模型”的方案重建权限体系：
- 简洁，便于统一治理
- 与 `gateway` 的 `apikey` 鉴权边界清晰
- 支持后续扩展到动态菜单、按钮控制和更细粒度授权
- 适合当前项目“允许推倒重建、不保留历史包袱”的前提
