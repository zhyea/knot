# 模型路由规则模块技术方案

## 1. 架构设计
- 子模块：规则编排、规则匹配、策略执行、命中日志。
- 组件：
  - `RouteRuleCompiler`：将规则表达式编译为可执行 AST。
  - `RouteEngine`：按优先级匹配并执行策略。
  - `FallbackExecutor`：故障转移与降级。

## 2. 数据设计
- 核心表：`routing_rules`。
- 推荐新增：`routing_rule_versions`。

## 3. 接口设计
- `GET/POST/PUT /api/routing-rules`
- `POST /api/routing-rules/{id}/test`

## 4. 关键流程
- 规则发布：语法校验 -> 冲突检测 -> 版本化发布 -> 缓存刷新。
- 运行匹配：读取规则缓存 -> 条件匹配 -> 策略执行 -> 通过插件事件发布请求记录与路由切换事件。

## 5. 异常与性能
- 当规则引擎异常时启用默认兜底路由。
- 支持热更新缓存，不重启服务生效。

## 6. 插件事件设计
- 网关核心不内置请求记录表和切换日志表，避免把运行日志存储方案固化在主库中。
- 请求完成后发布 `gateway.request.completed` 事件，载荷包含 `requestId`、`consumerId`、`ruleId`、`targetType`、`targetId`、`modelCode`、`status`、`latencyMs`、`errorCode`、`inputTokens`、`outputTokens`、`totalTokens`、`costSnapshot`、`occurredAt`。
- 发生主目标失败并尝试备用目标时发布 `gateway.routing.switched` 事件，载荷包含 `requestId`、`ruleId`、`consumerId`、`fromTargetType`、`fromTargetId`、`toTargetType`、`toTargetId`、`reason`、`priorityOrder`、`occurredAt`。
- 日志增强类插件订阅上述事件后自行决定推送到外部日志平台、消息队列、时序库或自定义业务库。
- 插件执行默认 `fail-open`，插件推送失败只记录插件执行错误，不影响网关请求主流程；需要强一致审计时再为特定插件启用 `fail-close`。
