# 实施计划与迁移建议

## 1. 总体判断

如果按新项目重建，建议把现有插件实现视为参考样例，而不是迁移基座。

现有可复用的是“接口雏形”，不是“插件系统本身”。

## 2. 建议实施顺序

### Phase 1：统一扩展点模型

目标：

- 定义插件核心模型
- 定义扩展点编码
- 定义插件实例与绑定模型

交付：

- 核心 SPI
- 插件目录服务
- 插件绑定解析器

### Phase 2：内置插件运行时

目标：

- 使用 Spring Bean 作为第一批插件来源
- 打通请求链路调度器

优先打通：

- `UsageExtractor`
- `ScheduledTaskHandler`
- `UpstreamProviderAdapter`

### Phase 3：请求链插件化

目标：

- 在网关主链路插入标准扩展点

建议优先阶段：

- `gateway.pre-route`
- `gateway.pre-upstream`
- `gateway.post-upstream`

### Phase 4：管理台与数据模型

目标：

- 重做插件管理数据结构
- 重做页面

### Phase 5：外部插件包

目标：

- 目录扫描
- manifest
- ClassLoader 隔离

## 3. 首批应该插件化的能力

建议优先做以下 6 类：

1. Usage 解析器
2. 定时任务处理器
3. 供应商接入适配器
4. 路由评分策略
5. 通知渠道
6. 请求前后过滤器

## 4. 与现有代码的迁移建议

### 可直接保留接口思路

- `UsageExtractor`
- `ScheduledTaskHandler`
- `UpstreamProviderAdapter`
- `UpstreamProtocolExecutor`

### 建议废弃或重做

- 现有 `plugins` 表
- 现有 `PluginService` 和 `PluginController` 的管理模型
- 现有 `plugin_type=PRE/POST` 的分类方式

## 5. 风险与注意事项

### 风险一：过度插件化

如果把核心模型也做成插件，系统会失控。

应对：

- 只开放明确边界
- 所有插件必须声明能力

### 风险二：插件失败拖垮主链路

应对：

- 超时
- 熔断
- fail-open / fail-close

### 风险三：插件版本不兼容

应对：

- manifest 兼容矩阵
- 核心 SPI 版本号

## 6. 建议的首个落地里程碑

如果只做一个最小可用版本，我建议交付：

1. `PluginRuntime`
2. `PluginBindingResolver`
3. `GatewayFilterPlugin` SPI
4. `UsageExtractorPlugin` SPI
5. `ScheduledTaskPlugin` SPI
6. 新版插件表结构

这一步完成后，项目就真正有“插件系统”了，而不只是“插件管理页面”。
