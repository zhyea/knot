# 网关路由处理重构变更日志

## 1. 变更日期
- 2026-05-31

## 2. 变更背景
- 网关请求处理原先集中在 `service` 包中，路由解析、请求编排、供应商适配、频控额度校验等职责边界不够清晰。
- 网关运行时逻辑需要从传统 `service` 层中拆出，`service` 层保留为数据库访问和领域内专属操作的承载层。
- 路由请求在失败切换时会遍历多个候选模型，频控和额度校验需要区分请求级资源和目标级资源，避免重复校验或提前终止。

## 3. 主要调整

### 3.1 网关分层调整
- `controller`：保留 HTTP 入口职责，负责接收 header、请求体和协议类型。
- `runtime`：承载网关请求模板、上下文、请求编排和失败切换流程。
- `routing`：承载路由解析逻辑，包括消费者、路由规则、应用、模型和模型池候选目标解析。
- `upstream`：承载请求适配、供应商协议适配和上游代理调用。
- `traffic`：承载频控和额度校验。
- `billing`：承载消耗明细和计费计算。
- `service`：保留为数据访问和领域服务层，目前通过 `GatewayDataService` 对接 Mapper、缓存数据库配置并向运行时组件提供查询能力。

### 3.2 Header 校验与异常处理
- `GatewayController` 中 `Authorization`、`Rule`、`traceparent` 均使用 `@RequestHeader(required = true)`，缺失 header 由 Spring MVC 统一抛出 `MissingRequestHeaderException`。
- `AbstractGatewayRequestTemplate` 中的 header 校验只处理值是否合法，例如 `Authorization` 格式、空白 `Rule`、空白 `traceparent`。
- 新增网关专属异常：
  - `GatewayRequestException`
  - `GatewayAuthException`
  - `GatewayInvalidRequestException`
- 新增错误响应实体 `GatewayErrorResponse`，替代在异常处理器中手工拼接 `Map`。
- `GatewayExceptionHandler` 改为通过 `@ResponseBody` 和 `@ResponseStatus` 固定返回 HTTP 状态和错误响应，不再手工构造 `ResponseEntity.status(...)`。

### 3.3 频控与额度校验流程
- 一次网关请求只会对应一个消费者、一个应用、一个路由规则，这三类资源在进入模型遍历前只校验一次。
- 一次网关请求可能对应多个候选模型，模型级校验在遍历候选模型时逐个执行。
- 一个供应商可能提供多个候选模型，因此在一次请求内增加 `TrafficCheckContext`，缓存供应商校验结果，避免同一供应商重复校验和重复计数。
- 应用、消费者、路由规则任意一个校验失败，直接终止本次请求。
- 当前模型或供应商校验失败时，不终止整个请求，继续尝试下一个候选模型。
- 上游请求失败后，继续尝试下一个候选模型，并在请求前重新执行目标级校验。

### 3.4 错误语义
- `RATE_LIMIT_EXCEEDED` 映射为 HTTP `429`。
- 新增网关错误类型 `rate_limit_error`。
- 上游失败仍按上游错误处理，不与频控/额度拒绝混用。

## 4. 关键类调整
- `GatewayController`：只负责 HTTP 入口和协议分发。
- `GatewayRequestHandler`：网关请求编排、路由级校验、目标遍历、失败切换。
- `AbstractGatewayRequestTemplate`：模板方法入口和通用请求流程。
- `RoutingResolver`：解析消费者、规则、应用和候选目标。
- `GatewayTrafficGuard`：拆分请求级校验与目标级校验。
- `GatewayDataService`：集中对接 Mapper 和数据库配置缓存。
- `UpstreamProxyClient`：构造上游请求上下文并委托协议处理器和供应商适配器。
- `GatewayBillingCalculator`：计算消耗明细和费用明细。

## 6. 后续注意事项
- 当前额度校验入口已按资源层级预留，但仍需要接入实际用量累计数据后才能做严格额度扣减和拒绝。
- 本节点频控已经具备基础窗口计数能力，多节点部署时需要进一步接入 Redis 或其他共享计数存储。
- 插件化请求记录和路由切换日志后续接入时，应复用当前请求上下文和目标切换流程，避免重新侵入主流程。
